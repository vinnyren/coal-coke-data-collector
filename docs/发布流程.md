# 发布流程

> 适用于本仓库的每次版本发布。本仓惯例为**直接提交 `main`**（无 feature 分支 / PR），
> 因此通用的分支型发布工作流（如 gstack `/ship`）的分支前置不适用；以本文为准。

## 一览

```
测试全绿 → 判定版本级别 → 写 CHANGELOG → 对齐文档版本号
→ 分块提交 → push（分叉则 rebase 保线性）→ bash scripts/release.sh
```

## 步骤

### 1. 前置验证

```bash
.venv/bin/python -m pytest -q     # 必须全绿才继续
git status --short                # 确认没有无关改动混入
```

### 2. 判定版本级别（四段式 `MAJOR.MINOR.PATCH.MICRO`）

对照历史量级判级，写入 `VERSION`（无换行）：

| 级别 | 何时用 | 历史参照 |
|---|---|---|
| MINOR | 新能力/新形态 | 0.3.0.0 一键安装、0.4.0.0 技能化 |
| PATCH | 小功能、脚本/流程工具 | 0.2.1.0 文档与注释完善 |
| MICRO | 琐碎修正 | 文案、注释级别改动 |

### 3. 写 CHANGELOG 条目

在 `CHANGELOG.md` 顶部插入（格式必须严格遵守，`release.sh` 依赖它提取发布说明）：

```markdown
## [X.Y.Z.W] - YYYY-MM-DD

一行摘要（可用全角冒号引出副题，冒号前的部分会成为 Release 标题）。

### 新增 / 修复 / 文档 / 测试

- 条目……
```

- **首个非空行**是摘要：`release.sh` 取全角冒号"："前的部分（无冒号则整行，去句尾句号）
  拼成 Release 标题 `vX.Y.Z.W — <摘要>`。
- 条目内容到下一个 `## [` 为止，整段会成为 GitHub Release 的正文。

### 4. 对齐文档版本号

- `docs/安装与使用指南.md` 头部的"版本 X.Y.Z.W"行与新版本一致。

### 5. 分块提交（可二分）

每个提交一个逻辑单元，`VERSION` + `CHANGELOG.md` 永远在**最后一个**提交：

```
feat/fix/docs: <各逻辑单元>
chore: bump version and changelog (vX.Y.Z.W)
```

### 6. 推送

```bash
git push origin main
```

被拒（远程有新提交）时：先 `git fetch origin`，确认远程增量内容，
用 `git rebase origin/main` 保持线性历史后重推。**禁止 force push。**

### 7. 打 tag + GitHub Release（脚本化）

```bash
bash scripts/release.sh
```

脚本行为（幂等，可重复执行）：

- 校验 `VERSION` 四段式格式，校验 `CHANGELOG.md` 存在对应条目（缺失即失败）；
- 校验本地 HEAD 与 `origin/main` 一致（未推送先拒绝）；
- 同名 Release 已存在则跳过并打印 URL；否则以 CHANGELOG 条目为正文创建
  `vX.Y.Z.W` tag + Release。
- 预演：`COAL_RELEASE_DRYRUN=1 bash scripts/release.sh` 只打印 tag/标题/正文，不做任何远程操作。

发布后 `git fetch origin --tags` 把 tag 同步回本地。

## 常见错误

| 报错 | 原因 | 处理 |
|---|---|---|
| `VERSION 格式非法` | VERSION 不是四段式 | 修 VERSION 文件 |
| `CHANGELOG.md 中找不到条目` | 忘写变更日志 | 回到步骤 3 |
| `本地 HEAD 与 origin/main 不一致` | 没推送或落后远程 | 回到步骤 6 |
| `需要 GitHub CLI` | 未装 gh 或未登录 | `brew install gh && gh auth login` |
