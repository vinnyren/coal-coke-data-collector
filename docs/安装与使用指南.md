# 煤焦交易数据采集技能 — 安装与使用指南

> 版本 0.2.0.0 · 适用于人工手动运行与 OpenClaw 类智能体无人值守定时运行。

采集**焦煤(jm)、焦炭(j)、动力煤(zc)**三个品种的**期货与现货**数据，幂等写入本地 **SQLite**，支持**历史回补**与**每日增量**。主通道为开源的 [AKShare](https://akshare.akfamily.xyz/)，另有多个免登录公开现货源作为补充。

---

## 1. 能力概览

| 类别 | 采集内容 | 数据表 |
|---|---|---|
| 期货 | 历史日线 / 主力连续 | `futures_daily` |
| 期货 | 实时行情快照 | `futures_realtime` |
| 期货 | 现货价与基差 | `spot_basis` |
| 期货 | 持仓排名 | `position_rank` |
| 期货 | 交易所库存 | `inventory` |
| 现货 | 分地区现货/指数价（结构化：品种×地区类型×地区） | `spot_regional` |
| 现货 | 原始指数价（未分类留存） | `index_price` |
| 统计 | 跨地区统计（均价/极差/最低最高地 + `ALL` 汇总） | `spot_regional_stats` |

所有表均以业务主键唯一约束，**重复运行幂等去重**（`INSERT ... ON CONFLICT`）。表结构见 `db/schema.sql`。

## 2. 环境要求

- **Python ≥ 3.9**（仅用标准库 + 下列第三方包，无需打包/虚拟环境也可运行，但推荐用 venv 隔离）。
- 联网（AKShare 与各现货源均需访问公网）。
- 磁盘：SQLite 库与 `runs/` 报告目录，长期运行建议留几百 MB。

第三方依赖（`requirements.txt`）：

| 包 | 版本 | 用途 |
|---|---|---|
| akshare | ≥1.12 | 主数据通道（期货行情/基差/持仓/库存） |
| pandas | ≥1.5 | AKShare 返回的数据帧处理 |
| requests | ≥2.28 | 现货公开源 HTTP 抓取 |
| beautifulsoup4 | ≥4.11 | 现货页面 HTML 解析 |
| pytest | ≥7.0 | 测试（运行采集本身不需要） |

## 3. 安装

```bash
# 1) 进入项目目录
cd /path/to/煤炭和焦炭交易数据采集技能

# 2)（推荐）建虚拟环境
python3 -m venv .venv && source .venv/bin/activate

# 3) 安装依赖
pip install -r requirements.txt

# 4) 冒烟验证（跑一小类，几秒钟）
python run.py --mode daily --kind inventory --format text
```

看到 `采集完成 ... exit_code=0` 即安装成功。首次运行会自动在 `db/coal_data.db` 建库建表。

## 4. 快速开始

```bash
# 首次：历史回补（拉全历史日线/基差等，耗时较长）
python run.py --mode backfill --start 2015-01-01

# 日常：每日增量（只补最新交易日，快）
python run.py --mode daily
```

- **回补**只需做一次；之后每个交易日跑一次**增量**即可。
- 首次回补建议分类别跑（见 `--kind`），便于观察进度、隔离失败。

## 5. 命令行参数

```
python run.py [--mode backfill|daily] [--kind all|futures|spot|rank|inventory|regional]
              [--start YYYY-MM-DD] [--format json|text]
```

| 参数 | 取值 | 默认 | 说明 |
|---|---|---|---|
| `--mode` | `backfill` / `daily` | `daily` | 回补拉全历史；增量只补最新交易日 |
| `--kind` | 见下表 | `all` | 只跑某一类采集器 |
| `--start` | `YYYY-MM-DD` | `2015-01-01` | 仅 `backfill` 生效，历史起点 |
| `--format` | `json` / `text` | `json` | stdout 输出格式（见 §7） |

`--kind` 取值：

| 值 | 包含的采集器 |
|---|---|
| `all` | 下列全部 |
| `futures` | 历史日线 + 实时行情 |
| `spot` | 现货价与基差 |
| `rank` | 持仓排名 |
| `inventory` | 交易所库存 |
| `regional` | 生意社 / CCTD / 全国煤炭交易中心 / 中国太原煤炭价格指数 + 跨地区统计 |

示例：

```bash
python run.py --mode daily --kind futures            # 只更期货
python run.py --mode daily --kind regional           # 只更现货多地与统计
python run.py --mode backfill --kind regional        # 回补现货历史（含 ctctc 周度历史）
```

## 6. 无人值守 / 定时运行

设计目标之一是**可被调度器或智能体无人值守周期运行**。核心约定：

### 退出码（最可靠的健康信号，契约恒为 `{0, 2, 3}`）

| 退出码 | 含义 | 处置 |
|---|---|---|
| `0` | 全部跑通（含当日 0 行的软失败） | 健康 |
| `3` | 有采集器异常（报告内 `status=error` 标明） | 部分失败，看 `results[].error` |
| `2` | 致命（DB 初始化 / 报告写出失败 / 任何逃逸异常） | 需人工介入，看 `error` 字段 |

> 程序在启动时已把 stdout 重配为 UTF-8，并在 `main()` 顶层兜底任何异常，确保退出码始终落在 `{0,2,3}`，不会因中文/无 locale 崩成 exit 1。

### crontab（每个交易日 17:30，收盘后）

```cron
30 17 * * 1-5 cd /PATH/TO/项目 && flock -n /tmp/coal.lock /usr/bin/python3 run.py --mode daily >> logs/cron.log 2>&1
```

- **推荐 17:30（CST）而非凌晨**：`daily` 以 UTC 当日为起点，凌晨（00:00–08:00 CST）会落到 UTC 前一日（幂等 upsert 下仅多回补一天，无正确性问题）。
- `flock -n` 防止上一轮未跑完与下一轮重叠（重叠会触发 SQLite `database is locked`，被误报为 exit 3）。
- 无 locale 的环境建议加 `LANG=C.UTF-8`（程序已有 UTF-8 兜底，此为双保险）。

完整 crontab 与 macOS launchd 示例见 `scripts/cron.example`。

### OpenClaw 类智能体调度

让调度器按周期派生会话，指令示例（不要交互式提问，自动完成并回报）：

> 在项目目录运行 `python run.py --mode daily --kind all`，然后读取 `runs/latest.json`，用中文总结各源 `status/rows`、`totals` 与退出码。若退出码非 0 或存在 `status=error`，明确标注"采集告警"并列出失败源与 error 摘要。

完整示例见 `scripts/openclaw-task.example.md`。

### 报告目录保留

每次运行写一个 `runs/run-<UTC时间戳>.json` 归档，长期累积。定期清理示例（保留 30 天）：

```bash
find "${COAL_RUNS_DIR:-runs}" -name 'run-*.json' -mtime +30 -delete
```

## 7. 运行报告

每次运行都会：

1. 向 **stdout** 打印一份报告（`--format json` 为单个可解析 JSON；`--format text` 为人类可读摘要）。
2. 写 **`runs/latest.json`**（权威机器输出，原子覆盖，含致命失败也写出）。
3. 写 **`runs/run-<UTC时间戳>.json`** 归档（同秒冲突自动追加 `-N`，不覆盖）。

> ⚠️ 解析优先用 `runs/latest.json`：采集库偶发向 stdout 打印可能干扰 JSON，而 `latest.json` 是原子写、结构稳定的权威来源。

报告字段：

```jsonc
{
  "started_at":  "2026-07-03T09:30:00+00:00",   // ISO8601 UTC
  "finished_at": "2026-07-03T09:31:05+00:00",
  "duration_ms": 65000,                          // 顶层墙钟耗时（钳位 ≥0）
  "mode": "daily",
  "kind": "all",
  "results": [                                   // 每个采集器一条
    {"name": "inventory", "status": "ok", "rows": 112, "error": null, "duration_ms": 7156}
  ],
  "totals": {"rows": 112, "ok": 1, "empty": 0, "error": 0},
  "exit_code": 0
}
```

- `status` 取值：`ok`（跑通且行数>0）/ `empty`（跑通但 0 行，软失败）/ `error`（fetch 抛异常）。
- **顶层 `error` 字段仅在致命失败（`exit_code=2`）时出现**，为一段 error 摘要；成功/部分失败的报告不含该键。逐采集器的错误在 `results[].error`。

## 8. 环境变量

| 变量 | 作用 | 说明 |
|---|---|---|
| `COAL_DB_PATH` | 指定 SQLite 库位置 | 支持 `~`；未设置回退 `db/coal_data.db` |


## 9. 数据来源与容错

- **主通道 AKShare**（开源、合法、零成本）：聚合新浪/东财/交易所的期货行情、基差、持仓、库存。
- **补充现货源**（`sources/`，可插拔，单源失败不影响主数据）：
  - `web_cctd`：CCTD 煤价指数中心公开指数页，双写 `index_price`（原始）与 `spot_regional`（可分类）。
  - `web_100ppi`：生意社现货表全国价，过 `HW_CHECK` JS 反爬挑战。
  - `web_ncexc`：全国煤炭交易中心 JSON 接口（直达煤=产地、下水煤=港口）。
  - `web_ctctc`：中国太原煤炭价格指数 JSON 接口，含 2023 至今周度历史。
- **失败隔离**：每个采集器在 `BaseCollector.run()` 内独立捕获异常并标 `status=error`，不中断其它采集器；网络类调用带 `with_retry` 重试。

## 10. 常见问题排错

| 现象 | 可能原因 | 处理 |
|---|---|---|
| `exit_code=3`，某源 `status=error` | 网络波动 / 上游接口变动 / 反爬 | 看 `results[].error`；单源失败不影响其它，可重跑该 `--kind` |
| `exit_code=2`，`error` 含 `build_store failed` | DB 路径不可写 / 磁盘满 | 检查 `COAL_DB_PATH` 目录权限与磁盘 |
| `exit_code=2`，`error` 含 `write_report failed` | `runs/` 目录不可写 | 检查 `COAL_RUNS_DIR` 权限 |
| 某源 `status=empty`（0 行） | 当日无新数据 / 非交易日 | 通常正常（软失败），持续多日为空再排查 |
| SQLite `database is locked` | 两次运行重叠 | 用 `flock` 串行化（见 §6） |
| cron 下中文乱码/崩溃 | 环境无 UTF-8 locale | 程序已兜底；仍建议设 `LANG=C.UTF-8` |
| 完整异常堆栈在哪看 | 报告只留限长摘要 | 见 `logs/collector.log`（含完整 traceback） |
